# ArchAgent 技术方案

详细编辑器边界见 [ArchAgent-编辑器设计方案.md](./ArchAgent-编辑器设计方案.md)。

## 技术栈

| 层级 | 方案 | 职责 |
| --- | --- | --- |
| 桌面容器 | Electron + electron-vite | 主进程、预加载桥与窗口生命周期 |
| 编辑渲染 | Three.js + React Three Fiber | 单一 WebGL 视图、建筑几何、资产预览与选择 |
| 相机 | Drei `CameraControls` + 自研导航球 | 平移、缩放、预设与导航球旋转 |
| 场景领域 | `SceneSnapshot` + `SceneCommand` | 可验证节点、历史与项目持久化 |
| 三维交换 | Three exporters/loaders + Main 服务 | GLB/GLTF/OBJ/STL 导入导出与受控文件路径 |
| 智能体 | Agent 工具 + 混元 3D 服务 | 场景命令、资产生成、导入与布局 |

## 运行时原则

1. `SceneService` 是唯一权威状态；R3F 只消费快照。
2. 主线不依赖 WebGPU，也不维护第二套 Pascal 或兼容渲染器。
3. 网格、导航球、选择描边等编辑辅助物不参与模型导出。
4. 外部模型文件进入项目受控资产目录；Renderer 不接收任意本机路径。
5. 性能优化优先采用按需加载、预览图缓存、几何复用、实例化和 LOD，而不是额外 Canvas。

## 参考资料重建编排

户型图、平面图和实物照片的重建不由单轮对话直接触发昂贵的 3D 生成。Agent 先读取资料、检索全局构件库并创建可持久化的重建计划；计划可包含事实、假设、候选资产、待生成资产和必答问题。只有用户完成必答选项并点击“按当前方案生成”后，编排器才会执行。

```text
资料分析 → 资产复用/拆分计划 → 必答选项 → 用户确认
                                      ↓
                         并行低模生成/入库 → 按计划逐件摆放
```

- 计划绑定 `workflowId + revision`；新方案、资料或选项变化会要求重新确认，旧版本不能继续启动任务。
- 复用资产和生成资产都只在确认后放置。生成任务默认调用供应商 `LowPoly` 配置；同类多件实例只生成一次，再按多个变换摆放。
- 工作流随会话状态持久化；应用重启时会将中断中的任务重新排队。任务失败不会覆盖已完成资产，计划进入“需要处理失败项”。

## 会话资源系统

所有用户上传文件、Agent 生成文件与派生文件进入同一会话资源库，不再以 `attachment_id` 和 `artifact_id` 两套标识隔离。每项资源拥有稳定的 `resource_id`，文件路径仅在主进程内部保存，模型和工具不得传递或猜测本机路径。

资源注册表持久化在 `<project>/.agent/resources.json`，与会话状态文件 `<project>/.agent/sessions.json` 分离；同一项目的所有会话共享资源本体，会话只保存各自的引用状态。

```text
用户上传 ───────┐
工具生成 ───────┼─> 会话资源库（resource_id） ─> 上下文选择器 ─> 多模态模型
工具派生/裁剪 ──┘             │                       │
                              ├─> 其他工具输入         └─> 当前轮附图/文本/数据摘要
                              └─> send_file 用户交付
```

### 资源契约

```ts
interface SessionResource {
  id: string; // resource_<uuid>，唯一公开标识
  sessionId: string;
  kind: "image" | "document" | "table" | "database" | "model3d" | "text" | "archive" | "other";
  name: string;
  mimeType: string;
  size: number;
  source: "user_upload" | "generated" | "derived";
  parentResourceIds: string[];
  path: string; // 仅主进程可见
  metadata: Record<string, unknown>; // 页数、表结构、模型包围盒等
  status: "ready" | "processing" | "failed";
  confirmed: boolean;
  pinned: boolean;
  createdAt: string;
}
```

- 用户上传、剪裁图、提取图、效果图、渲染图、GLB 与报告都必须登记资源；派生资源保留 `parentResourceIds`，形成可追溯关系。
- 每个工具统一接收 `input_resource_ids`，返回 `output_resource_ids`；`send_file` 接收 `resource_id`。业务参数仍可保留明确字段，例如重建计划中的 `source_resource_ids`。
- Agent 每轮收到资源清单（ID、名称、类型、来源、父资源、状态、确认状态、可用摘要），从而可稳定指代“刚才效果图”或“沙发裁剪图”。
- 不向模型暴露内部路径、二进制内容或 API 密钥；路径解析、MIME 校验和访问控制均由主进程完成。

### 按类型建立可用表示

| 资源类型 | 保存内容 | 为模型与工具提供的表示 |
| --- | --- | --- |
| 图片 | 原图及尺寸/哈希 | 原图多模态块、视觉工具输入 |
| PDF、DOCX、TXT、Markdown | 原文件、文本索引、页/段落定位 | 相关片段；版式重要时渲染关键页图片 |
| CSV、XLSX、JSON | 原文件、字段、统计与抽样 | schema、样本、受限查询结果；不注入整张大表 |
| 数据库 | 连接或快照、schema | 表结构和受限查询结果 |
| GLB、OBJ、STL | 原模型、包围盒、材质和资产库关联 | 自动渲染预览图并附元数据；二进制只交给导入/导出工具 |
| ZIP、未知二进制 | 原文件与检测状态 | 默认不注入；先解包或检查再产生派生资源 |

### 每轮上下文选择

开启图片输入时，运行时而非模型在发起请求前选择需要附带的资源，防止模型还未看到图片就无法选择图片。选择顺序如下：

1. 本轮新上传的图片与用户明确点名的 `resource_id` 必带。
2. 当前重建计划绑定的原始参考图、已确认裁剪图和等待用户确认的最新效果图优先带。
3. 根据用户当前消息、资源名称、父子关系和工作流阶段匹配的图片按相关度补充。
4. 达到图片数量、像素和 token 预算后停止；其他资源保留在清单中，Agent 可明确请求或调用读取工具。

工具刚生成图片时，运行时先登记 `resource_id`，再把图片作为该工具后续模型回合的多模态结果附回，确保 Agent 能看到效果图而不是只读到“生成成功”。历史图片不应每轮无限重复发送；`pinned`、`confirmed`、当前工作流和用户引用决定其是否重新附带。

### 工具结果触发的视觉回注

工具调用结束后，Agent 仍会在同一轮继续推理，因此资源选择必须发生在“工具结果回送模型”之前。触发分为三类：

1. **自动回注**：生图、提取图、裁剪图、模型渲染预览等视觉工具成功后，运行时登记输出资源，并将结果组装为 `[{ type: "text" }, { type: "image" }]` 回给模型。该图片成为下一次模型推理的工具结果内容，Agent 可立即检查并决定继续、修改或请求用户确认。
2. **首轮自动选图**：收到用户消息后、首次模型请求前，上下文选择器基于本轮上传、当前工作流、确认状态和消息引用生成 `ContextMediaPlan`，把选中图片作为本轮 user prompt 的图像块。
3. **模型主动取图**：资源清单只展示 ID 与摘要；当 Agent 判断需要查看未附带的历史图片时，调用 `view_resources(resource_ids)`。该工具按预算读取资源并以图像内容返回，下一次同轮模型推理自然获得图片，无需暴露路径或重启会话。

`ContextMediaPlan` 由运行时维护，并记录每张图的选择原因、token/像素预算和已附带轮次。自动回注的工具输出优先级最高；若超出预算，保留工具输出和用户显式指定图，淘汰低相关历史图。非图片资源不以二进制回送：文档返回片段，表格返回 schema/查询结果，3D 模型返回预览图与元数据。

### `view_resources` 统一查看工具

`view_resources` 是 Agent 主动读取历史资源的唯一高层入口。它接收公开资源 ID，不接受路径，并按资源类型返回对当前任务最有用的模型内容。

```ts
interface ViewResourcesInput {
  resource_ids: string[];
  purpose: string; // 为什么需要查看，用于选择转换策略与预算
  query?: string; // 文档、表格或数据库的关注主题
  page_range?: [number, number]; // PDF/DOCX 的可选页范围
}
```

| 资源类型 | `view_resources` 行为 | 回送模型的内容 |
| --- | --- | --- |
| 图片 | 在尺寸与像素预算内读取原图 | `ImageContent`，模型直接可见 |
| PDF、DOCX、TXT、Markdown | 文本提取、按 `query` 检索片段；版式是任务关键时生成指定页预览 | 带页/段落定位的文本，必要时附页图 |
| CSV、XLSX、JSON、数据库 | 提取 schema、统计、样本或受限查询结果 | 结构化文本；不返回整份大数据 |
| GLB、OBJ、STL | 读取格式、尺寸、包围盒、材质与资产关联，并默认生成预览图 | 预览 `ImageContent` + 元数据文本 |
| ZIP、未知二进制 | 识别类型、列出安全可读条目；必要时产生派生资源 | 检查摘要及派生 `resource_id` |

图片和 3D 模型预览必须作为工具结果的 `ImageContent` 返回，而非把 URL、Base64 或本机路径拼到文本中。文档与数据文件同样不能无差别全文注入：`purpose` 和 `query` 决定转换器返回的范围，运行时继续执行大小、页数、行数和 token 预算限制。

### 迁移与边界

1. 将现有附件与产物记录迁移为 `SessionResource`；保留原 ID 仅用于一次性迁移映射，不再暴露给 Agent 工具。
2. 删除独立视觉识别入口；主 Agent 通过上下文选择器直接看到原图，其他资源统一由 `view_resources` 按类型读取。
3. 设计预览、物件提取、文生/图生 3D、脚本输出和导出都必须返回资源 ID；后续工具只能以资源 ID 作为文件输入。
4. 文本、表格和模型资源不伪装为图片。只有显式生成的页渲染图或模型预览图才进入视觉上下文，并保留其父资源关系。
5. 资源清理必须同时移除派生索引和会话引用；删除父资源前提示其仍被工作流、资产或子资源引用。
