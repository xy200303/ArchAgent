# 参与贡献

感谢你为 ArchAgent 做出贡献。本指南适用于代码、测试、文档、设计规格和构件资源相关的所有改动。

## 开始前

1. 阅读根目录的 [README.md](README.md) 和与任务相关的 `docs/design/`、`specs/active/` 文档。
2. 确认改动范围、预期行为和验收方式；涉及不明确的产品规则、数据兼容性或安全边界时，先与维护者确认。
3. 从最新目标分支创建聚焦的工作分支。一个分支和一个 PR 应只解决一个相互关联的问题。

## 本地开发

环境要求见 README。首次安装和启动：

```bash
npm install
copy .env.example .env.local
npm run dev
```

`.env.local` 可能包含密钥，不得提交。不要在 Issue、PR、截图、日志或测试夹具中暴露 API Key、访问令牌、个人路径或用户数据。

## 代码与文档约定

- 保持现有目录边界：Main、Preload、Renderer 和 `shared/` 的跨进程契约不得混放。
- UI、领域逻辑、文件访问和持久化应保持职责分离；优先复用项目已有模式和依赖。
- 修改 IPC、场景契约、配置、资源访问或 Agent 工具时，检查对应的类型、测试和设计文档是否需要同步更新。
- 文档中的命令、路径、环境变量和工具名称必须来自实际实现，不能凭空补充。
- 避免无关的重命名、格式化、锁文件变更、构建产物和二进制文件。确有必要时，请在 PR 中说明原因。

## 测试与验证

每次提交前，至少运行与改动风险相匹配的验证。通常应运行：

```bash
npm run typecheck
npm test
git diff --check
```

UI 或 Electron 交互改动还应进行相应的手动验证，并在 PR 中记录实际操作、平台和结果。无法运行的检查必须如实注明原因，不得将未执行的命令标记为通过。

## 提交规范

- 提交前使用 `git status --short` 和 `git diff --name-status` 确认仅包含本次工作。
- 提交信息应简洁、具体且以动作开头，例如 `完善场景预览校验`；不要使用空泛的“更新代码”。
- 不要将不相关的改动混入同一提交，也不要改写或清除其他贡献者的未提交工作。
- 禁止使用 `git reset --hard`、`git checkout --` 等破坏性命令准备提交，除非维护者明确要求。

## Pull Request 流程

1. 先阅读并完整填写 [PR 模板](.github/PULL_REQUEST_TEMPLATE.md)。
2. 推送分支前再次检查变更和验证结果，只暂存属于该 PR 的文件。
3. PR 标题应简洁、动作导向，并与提交信息和 PR 正文使用相同语言。
4. 清楚说明变更、动机、验证、风险、兼容性影响和需要关注的代码路径；必要时附上 UI 截图或录屏。
5. 不要强制推送或重写他人提交，除非维护者明确授权。

维护者可能要求拆分过大的 PR、补充测试、更新文档或说明安全与兼容性影响。请在同一分支内回应这些反馈，并保持讨论聚焦于可验证的行为。
